企业内部已经积累大量需求、方案、代码仓库、产品版本、历史决策、测试结果、发布记录、线上日志和知识文档，但这些信息通常分散在文档、Issue、聊天记录、代码平台、CI/CD、监控日志和人工经验中。传统 AI 聊天工具只能回答局部问题，难以承载可配置、可审计、可复用的业务闭环，更难感知一个需求从提出到上线后反馈的完整研发上下文。

AI Brain 的产品方向是建设一个企业 AI 大脑平台，让不同业务领域可以配置自己的“业务大脑”。v1 系列以产品研发大脑为样板，分阶段跑通从研发需求到产品详细设计、技术方案、代码开发辅助、代码 Review、自动化测试、发布上线评估、上线后分析、人工确认、模拟回写、Bug 管理和知识沉淀的研发全链路闭环。AI 助手聊天工作台作为系统问答和能力编排入口，帮助用户用自然语言查询 AI Brain 当前产品配置、需求任务进展、代码仓库、模型网关状态和项目开发情况；下一阶段支持通过 `@` 显式引用知识库、业务对象、插件、AI 能力和定时任务，在助理内生成可审阅的配置/执行草案，并按登录用户保存最近对话，便于继续追问和回看历史。首批配置草案覆盖“每周用户反馈洞察抽取”和“代码仓库质量/安全/规范巡检”定时作业，GitHub/GitLab/邮箱官方插件连接，以及 GitHub/GitLab 代码巡检、邮箱通知发送等官方插件动作：助手只返回 `assistant_action_draft`，前端必须以待确认草案卡片展示标题、风险、作业、连接或动作核心配置；当用户明确要求 AI/大模型分析代码巡检结果时，作业草案必须使用 `ai_generated` 并自动带出可用模型、Agent 和 Skill。定时作业草案可应用到新增作业表单，保留 `{{last_full_week.*}}` 等动态变量映射；插件连接草案可应用到插件管理新增连接表单，带出官方插件、Endpoint、环境、认证方式、Params 和 Headers；插件动作草案可应用到插件管理新增动作表单，带出官方插件、连接、请求路径、Params/Headers 和结果映射，其中邮箱通知动作草案默认使用 `email_notifications` 写入目标并在草案卡片展示“邮件通知记录”；当用户要求配置代码巡检定时作业但缺少可用 GitHub/GitLab 连接或动作时，助手必须同步返回连接草案、动作草案和作业草案，并在作业草案展示前置草案 ID，避免用户拿到无法保存的半空作业配置；用户先确认保存前置连接或动作草案后，后续动作/作业草案必须能在当前浏览器会话内解析前置草案对应的真实资源 ID，减少重复选择和手工粘贴；用户确认保存前不得写入 `scheduled_jobs`、`plugin_connections`、`plugin_actions` 或触发外部调用，保存后必须在对应配置和审计 payload 中保留草案来源，便于追溯配置来源。

## v1 交付边界

v1 系列采用分阶段交付，而不是一次性交付完整平台愿景。本文中的 `v1 系列` 指 v1 MVP、v1.1 和 v1.2 的整体产品方向；`v1 MVP` 指第一版可验收交付；`v1.1` 和 `v1.2` 是后续增强阶段。所有能力仍属于产品方向，但实现和验收按下表分层推进：

| 层级 | 定位 | 必须包含 | 暂不阻塞 MVP 的能力 |
|------|------|----------|--------------------|
| v1 MVP | 跑通可演示的研发大脑核心闭环 | 本地账号、产品/版本/模块配置、GitLab/GitHub 代码库绑定、需求审批、产品详细设计任务、技术方案任务、GitLab MR / GitHub PR 代码 Review 任务、人工确认、知识检索、知识沉淀候选审核、模拟 Issue、Markdown 导出、代码 Review 报告内部归档、审计、Docker Compose 健康检查 | GitLab/Jenkins/线上日志真实运营采集、GitLab/GitHub 评论或审批回写、自动化测试、完整 Bug 生命周期、全主体双向流程感知、上线后分析 |
| v1.1 研发任务扩展 | 补齐研发执行阶段辅助 | 开发计划、自动化测试任务、AI 自动测试 Bug 建议、代码 Review 执行器扩展能力 | 发布风险自动聚合、线上日志趋势分析 |
| v1.2 研发运营扩展 | 建立产品级运营视图 | GitLab 指标、Jenkins 发布、线上日志、用户使用数据、用户反馈、Bug 管理闭环、采集运行记录、插件管理、定时系统作业、AI Agent/Skill 能力配置、首页 IT 团队看板、从需求出发的全流程感知和 AI 迭代规划建议 | 真实外部系统双向写回、多租户治理、自动部署 |

验收时优先判断当前阶段声明的能力是否完整闭环。v1 MVP 的验收以最小可验证闭环为准：登录 → 产品和 GitLab 配置 → 需求审批 → 产品详细设计 → 技术方案 → 内部 GitLab MR 快照 → Code Review 报告 → 人工确认 → 内部归档 → 知识沉淀候选 → 审计可查。后续阶段的能力可以在界面、API 或数据结构中预留入口；尚未接入真实数据源时只返回空状态或禁用态，不返回示例数据、占位统计或伪造结果，不作为 v1 MVP 阻塞验收。

### v1 MVP 实施切片

v1 MVP 的产品验收边界保持不变，但工程交付应拆成可独立验证的三个切片，避免把工作流、GitLab、知识检索和审计风险集中到一次交付：

| 切片 | 目标 | 功能闭环必交 | 导航/空状态必交 | 暂不包含 |
|------|------|--------------|----------------|----------|
| MVP-A 基础 + Git 输入闭环 | 验证需求到可确认方案，并具备 GitLab MR / GitHub PR 输入依赖 | 本地账号、产品/版本/模块配置、GitLab/GitHub 只读仓库绑定、GitHub PR 列表、MR/PR 预览、MR/PR diff 快照、需求创建与审批、产品详细设计任务、技术方案任务、人工确认、Markdown 导出、基础审计、Docker Compose 健康检查 | 任务中心展示代码 Review 创建入口，但可在 MVP-A 暂不生成正式 Review 报告 | code-review 执行器、Review 报告内部归档、知识沉淀审核、复杂看板 |
| MVP-B Git Review 闭环 | 验证 GitLab MR / GitHub PR 代码 Review 的可审计闭环 | `code_review` 任务、结构化 Review 报告、人工确认、内部归档、不回写 GitLab/GitHub、执行器失败处理、diff 超限处理和相关审计 | Bug 管理、研发运营看板、用户洞察/迭代规划展示空状态或禁用态 | GitLab 每日指标采集、MR/PR 评论或审批回写、代码自动修改 |
| MVP-C 知识与治理闭环 | 验证输出可沉淀、可检索、可追踪 | 知识文档导入、索引状态、权限过滤检索、知识沉淀候选审核、模拟 Issue 幂等生成、主体级审计 | 后续阶段入口保持真实空状态或禁用态 | 自动化测试、完整 Bug 生命周期、Jenkins/线上日志/用户使用数据采集 |

MVP-A/B/C 是实施顺序和验收拆分，不改变 v1 MVP 对外承诺。只有三个切片均通过时，才宣称 v1 MVP 完整完成。MVP-A 已要求具备内部 GitLab 只读集成依赖，用于在需求和技术方案阶段固化代码上下文；正式 AI Code Review 报告生成、执行器调用和内部归档在 MVP-B 验收。

### MVP 成功指标

v1 MVP 不只以功能可用为成功，还必须证明研发大脑能缩短评审链路并产生可采纳的工程判断。MVP 完成后至少跟踪以下指标：

| 指标 | 口径 | MVP 目标 |
|------|------|----------|
| 需求到可评审产品详细设计耗时 | 从需求审批通过到 `product_detail_design` 待确认 | 相比纯人工流程减少 30% 以上，或单个样例流程不超过 30 分钟 |
| 技术方案一次采纳率 | `technical_solution` 首次 approve 或 edited_approve 占比 | ≥ 60% |
| Code Review 报告采纳率 | `code_review` 报告经 approve 或 edited_approve 的比例 | ≥ 50% |
| Review 高风险问题有效率 | 人工确认后保留的 high/critical finding 占 AI 输出 high/critical finding 的比例 | ≥ 50% |
| 知识沉淀复用率 | 已批准知识沉淀在后续任务检索中被引用的比例 | MVP-C 后开始统计，首期只要求可追踪 |
| 审计可追踪率 | 演示路径中的关键写操作和高影响 AI 动作可在审计中查到的比例 | 100% |

如果上述指标在真实或准真实样例中长期无法达成，应优先收窄任务类型、优化输入上下文或调整确认流程，而不是继续扩展 v1.1/v1.2 能力。

### MVP 系统角色映射

v1 MVP 实现优先使用少量系统角色承载业务权限，后续阶段可再拆分测试负责人、发布负责人和 IT 管理者等精细角色：

| 系统角色 | 对应业务角色 | MVP 关键权限 | 非 MVP 权限 |
|----------|--------------|--------------|------------|
| `admin` | 平台管理员 | 用户和角色配置、业务大脑配置、产品/版本/模块/Git 资源维护、模型网关配置、查看审计和运行状态 | 代替业务负责人做最终产品决策 |
| `product_owner` | 产品负责人 | 创建/审批需求、确认产品详细设计、查看任务和知识沉淀、查看迭代规划建议空状态入口 | 发布确认、Bug 完整闭环处理 |
| `rd_owner` | 研发负责人 | 创建和启动 AI 任务、确认技术方案、创建 GitLab MR 快照、创建/确认 code_review 任务、查看内部 Review 报告 | 自动部署、GitLab 回写审批或合并 |
| `reviewer` | 研发负责人或指定 Reviewer | 预览 MR、创建 MR diff 快照、确认或修改后采纳 code_review 报告 | 维护产品主数据、审批需求 |
| `knowledge_owner` | 文档/知识维护者 | 导入知识文档、查看索引状态、审核知识沉淀候选 | 审批需求、确认技术方案 |
| `viewer` | IT 管理者、测试负责人、测试人员或只读参与者 | 查看授权范围内的产品、需求、任务、知识、审计摘要和真实空状态看板 | 写操作、人工确认、配置变更 |

MVP 阶段的 `tester`、发布负责人和 IT 管理者能力可以先映射为 `viewer` 加页面空状态；完整 Bug 流转、发布确认、运营看板下钻和用户洞察决策按 v1.1/v1.2 再扩展角色。系统角色目录必须明确每个角色的业务角色映射、可见入口和限制边界，并在角色管理、用户管理角色目录和知识权限选择中保持同源展示。

## 核心业务主体

v1 系列需要把产品、需求、AI 任务、Bug、知识中心、研发度量/看板和用户洞察/迭代规划作为可独立维护或独立查看的一等业务主体，而不是把所有能力都收敛到 AI 任务详情内。

| 主体 | 用户价值 | 独立维护能力 | 与其他主体的关系 |
|------|----------|--------------|------------------|
| 产品 | 为需求、任务和知识检索提供稳定业务上下文 | 产品、版本、模块、Git 资源、相关系统 | 需求必须归属产品和版本；任务生成时固化产品上下文；知识可按产品/系统归类 |
| 需求 | 承载业务问题、目标、约束和审批结论 | 创建、补充、审批、驳回、关闭、生成任务 | 审批通过后生成 AI 任务；需求保留原始业务事实和审批状态 |
| AI 任务 | 承载 AI 执行、人工确认、研发全链路辅助、回写、导出和审计聚合 | 启动、补充信息、确认、重试、取消、导出；任务类型覆盖产品详细设计、技术方案、代码开发辅助、代码 Review、自动化测试、发布上线评估和上线后分析 | 可由需求生成；引用需求快照和产品上下文；可产出模拟 Issue、Review 结论、Bug、发布风险评估和知识沉淀候选 |
| Bug | 承载 AI 自动测试和人工测试登记的问题闭环 | AI 自动登记、人工登记、分派、修复、验证、关闭；在 Bug 管理中展示并按迭代版本查询 | 归属于产品/版本/模块，可关联需求、AI 任务、测试用例、GitLab 提交、Jenkins 发布和线上日志 |
| 知识中心 | 支撑企业知识主动导入、检索、治理和沉淀 | 导入、索引、检索、权限、沉淀审核、归档 | 为任务提供检索上下文；接收任务沉淀候选；可独立运营 |
| 研发度量/看板 | 支撑产品维度的代码质量、提交情况、发布和线上运行分析 | GitLab 提交采集、代码质量审核、Jenkins 发布采集、线上运行日志分析、首页 IT 团队看板、全流程感知视图 | 按产品、版本、模块、人员、环境和时间窗口聚合，为需求、设计、开发、Review、测试、发布和线上反馈之间的追溯和风险感知提供统一视图 |
| 用户洞察/迭代规划 | 支撑基于真实使用和反馈的产品迭代决策 | 用户反馈收集、详情查看、处理和转需求；用户使用数据采集和 AI 迭代规划建议作为后端能力保留 | 关联产品规划、需求池、Bug、线上日志、发布记录和研发投入；反馈可由产品负责人确认后直接转正式需求，需求记录保留来源 |
| AI 助手 | 支撑用户用自然语言查询 AI Brain 系统信息、研发进展，并在权限内编排平台能力 | 系统问答、项目进展问答、配置状态问答、`@` 引用知识库/业务对象/插件/AI 能力/定时任务、助理内配置草案、运行草案确认、最近对话、历史消息、建议问题引导 | 基于服务端脱敏系统上下文读取产品、需求、AI 任务、Git 资源、知识库、插件、AI 能力和定时作业状态；聊天历史按登录用户隔离；所有写配置、插件试运行、定时任务运行、删除或停用动作必须生成草案并经有权限用户确认，不替代正式任务流、审批流、知识检索权限或审计边界 |

### PRD 草案与产品详细设计边界

本文中的“PRD 草案”和 `product_detail_design` 不是两个独立 MVP 任务类型：

- 需求创建和审批阶段只保存原始业务事实、目标、约束和审批结论；系统可以展示轻量需求评估或 PRD 草案预览，但不作为独立工作流产物阻塞审批。

- 后续技术方案、代码 Review、知识沉淀和导出均以已确认的 `product_detail_design` 输出为依据。

## 详细场景

1. 平台管理员完成平台初始化，配置默认业务大脑、用户角色、模型网关、相关系统和基础产品。
2. 产品负责人或管理员在产品管理中维护产品、模块和内部 GitLab/GitHub 项目绑定，并在需求交付的“迭代版本”入口维护产品迭代版本；迭代版本主状态为规划中、开发中、测试中和已发布，历史归档仅作为后台管理状态；任意版本状态都必须能查看当前版本需求清单。版本进入开发迭代后，版本级代码分支在迭代版本中维护，可为同一版本关联多个代码库的基准分支、开发分支、分支状态和创建来源；测试中、已发布和历史归档版本不能用于新需求排期或新开发任务，未绑定或无权限的 Git 项目不能用于 v1 MVP 代码 Review 任务。
3. 需求提交人进入需求管理列表，点击“新增需求”，填写需求标题、背景、目标、约束和所属产品；目标迭代版本可暂不选择，便于先进入需求池。
4. 系统创建需求实体，状态为 `submitted`。
5. 负责人审批需求；审批通过但未排期时状态为 `approved`，可在需求管理批量排期或在迭代版本页归集多条同产品需求，排入规划中或开发中的有效迭代版本后状态为 `planned`，驳回后状态为 `rejected`，必要时可关闭需求。
5a. 迭代版本推进必须走专用状态推进动作，先展示影响预览，再同步可推进需求状态：规划中推进开发中时将 `approved/planned` 需求推进到 `ready_for_dev`；开发中推进测试中时将版本内已进入交付链路的 `approved/planned/ready_for_dev/designing/developing/code_reviewing` 需求统一推进到 `testing`，避免版本已测试而需求仍停留在需求池、排期、设计或开发状态；测试中推进已发布时将 `testing/ready_for_release` 需求推进到 `released`，未完成需求必须延期、取消、关闭或阻止发布。
6. 已排期需求可以生成 AI 任务，系统把需求输入、需求快照和产品上下文写入 `ai_tasks.input_json`，并根据业务阶段选择任务类型；AI 任务推进时同步更新需求的设计、开发、评审、测试、发布和验收状态。
7. AI 任务类型包括 `product_detail_design`、`technical_solution`、`development_planning`、`code_review`、`automated_testing`、`release_readiness` 和 `post_release_analysis`。
8. 前端或调用方启动 LangGraph 工作流，AI 结构化输入、检索相关知识、评估信息是否充足。
9. 信息不足时，任务进入 `waiting_more_info`，提交人补充信息后任务回到 `draft`，由前端或调用方再次启动继续运行。
10. 产品详细设计任务生成详细 PRD、交互说明、页面字段、业务规则和验收标准，产品负责人确认后进入技术方案阶段。
11. 技术方案任务生成模块边界、接口设计、数据模型、风险依赖和研发拆解，研发负责人确认后进入开发计划阶段。
12. 代码开发辅助任务生成开发任务清单、代码变更建议、实现步骤和待修改文件建议；v1 系列不自动修改代码、不自动提交 PR。
13. v1 MVP 代码 Review 任务基于 GitLab Merge Request 或 GitHub Pull Request 元信息和 diff 快照、需求快照、技术方案和产品上下文，调用可插拔 code-review 执行器生成结构化 Review 报告；一期默认执行器对接 Claude Code `code-review` skill，本地联调或 Chat-only 上游场景下若未配置外部执行器命令且已有 active/default Chat 模型网关，应复用模型网关生成结构化 Review 报告。
14. Review 报告包含摘要、整体风险等级、问题清单、严重程度、文件路径、行号或范围、问题分类、修改建议、置信度和执行器元数据，进入人工确认后只能在 AI Brain 内部归档，不回写 GitLab/GitHub 评论、审批状态或合并状态。
15. 自动化测试任务基于需求、验收标准、技术方案和已有测试用例生成测试建议、分析测试结果，并可登记来源为 `ai_auto_test` 的 Bug。
16. 发布上线评估任务结合代码质量、测试结果、Bug 状态、Jenkins 发布记录和线上日志历史，输出上线检查清单、风险评估、回滚建议和是否可发布结论。
17. 上线后分析任务结合线上运行日志、核心业务事件、错误率、延迟和发布版本，输出健康报告、异常趋势、疑似回归 Bug 和优化建议。
17a. 模型网关或 code-review 执行器临时失败时，失败任务必须停在可排查步骤，并允许在修复配置或上游恢复后使用同一 task_id 重试，避免复制新任务导致需求链路割裂。
18. 负责人确认关键 AI 产出后，系统生成模拟 Issue、Markdown 方案、内部代码 Review 报告归档、Bug 或发布评估结果；v1 MVP 的代码 Review 只归档 AI Brain 内部报告和人工确认结论。
19. 系统生成知识沉淀候选，知识维护者审核后进入正式知识库。
20. 系统每日采集本地 GitLab 提交、代码质量审核结果、Jenkins 发布数据、线上运行日志、实际业务系统用户使用数据和用户反馈，并按产品、版本、模块、人员、环境、用户群体和时间窗口聚合；采集过程必须留下可审计运行记录。
20a. 平台支持把用户洞察、日志采集、研发运营指标采集、看板刷新和 AI 迭代建议生成配置为定时系统作业。定时作业只负责按计划触发和记录运行，不直接绕过业务写入服务；新增作业页必须提供每周用户反馈洞察抽取和代码仓库质量/安全/规范巡检等场景模板，场景模板由服务端模板目录统一提供，声明模板版本、默认作业 payload、推荐场景和资源选择规则，页面和 AI 助手都必须复用该目录，不得各自硬编码默认 cron、动态变量映射和结果动作；选择模板后自动带出产品、数据连接、AI 模型、Agent、Skill、知识引用、插件动作、默认调度和结果动作配置，减少从空表单手工理解 ID 与 JSON 的成本；新增和编辑作业弹窗必须以“数据连接 → AI 处理 → 知识引用 → 结果动作”四段式编排预览展示配置完整性、必填状态、核心资源名称和数据连接测试结果，让管理员保存前就能判断取数、AI 分析和写入链路是否完整；涉及 AI 的定时作业必须装配已启用的 Agent、Skill、模型网关和工具策略，并在每次运行时保存解析后的配置快照。
21. 用户使用数据包括活跃用户、功能访问、关键路径转化、功能停留、异常退出和低使用功能；用户反馈包括满意度、问题反馈、改进建议、投诉和人工标注标签。
22. AI 结合产品规划、需求池、用户使用趋势、反馈热度、Bug、线上日志、发布记录和研发投入，生成下阶段优先迭代需求建议。
23. AI 自动测试和人工测试登记的 Bug 进入 Bug 管理，关联产品、版本、模块、需求、任务、测试用例、GitLab 提交、Jenkins 发布或线上日志事件。
24. 首页 IT 团队看板展示需求、研发进展、Bug、线上系统健康、核心业务运行、用户使用趋势、用户反馈趋势、AI 迭代规划建议、发布状态、代码质量和知识沉淀统计。
25. 管理者可以从首页看板下钻到需求、任务、Bug、发布记录、代码质量、线上日志、用户使用明细、用户反馈明细和迭代规划建议，保留产品与时间范围上下文。
26. 系统形成软件研发全流程感知链路，把需求、产品详细设计、技术方案、代码提交、代码 Review、自动化测试、Bug、Jenkins 发布、线上日志、用户使用、用户反馈、迭代规划建议、知识沉淀和审计事件按产品、版本、模块、需求和任务串联。
27. 任务详情、需求详情、Bug 详情、首页看板和迭代规划建议详情均可展示当前主体关联的上游依据、下游影响、风险信号和下一步建议。
28. 用户进入 AI 助手聊天工作台，询问 AI Brain 系统配置或项目开发进展；助手基于服务端脱敏系统上下文回答产品、需求、AI 任务、Git 仓库和模型网关状态，并给出可继续追问的问题建议；系统按登录用户保存会话和消息，最近对话只展示本人历史。
28a. 用户在 AI 助手里通过 `@` 显式引用知识库、业务对象、插件动作、AI Agent/Skill、模型网关配置、定时作业或运行实例；助手可基于这些引用生成知识问答、配置草案或运行草案。任何 AI 能力配置、插件配置、定时作业创建/修改/删除、插件试运行和作业手动运行都必须先展示草案、影响摘要和风险提示，由有权限用户确认后才调用正式业务接口。
29. 全流程产生审计事件，可在运行审计中按任务、主体、用户、事件类型和时间查询。

## 功能范围

### In Scope

以下为 v1 系列范围；是否阻塞当前交付以“v1 交付边界”和“上线计划”中的阶段归属为准。

**v1 MVP 必交**:
- 本地账号和 Bearer Token 认证。
- 业务大脑配置读取，v1 默认研发大脑 `rd_brain`。
- 产品、迭代版本、产品模块、产品 Git 资源和迭代版本代码分支配置，用于提交需求、排期、开发任务、PR/Review 和代码 Review 时形成产品版本上下文；新增需求可先不指定迭代版本，进入版本开发后由迭代版本统一维护版本级分支。
- 需求实体列表、审批、驳回和审批后生成产品详细设计任务。
- 平台模型网关配置，用于维护 OpenAI-compatible 默认模型入口。
- AI 助手聊天工作台，用于回答 AI Brain 系统配置、产品、需求、任务、Git 仓库、模型网关状态和项目开发进展问题；该入口只依赖 Chat 模型网关，不要求 Embedding 可用，并按登录用户保存最近对话与历史消息。后续升级为可通过 `@` 引用知识库、业务对象、插件、AI 能力和定时任务的统一工作台。
- AI 任务创建、启动、失败后重新启动、补充信息、取消和详情查询。
- MVP AI 任务类型覆盖产品详细设计、技术方案和内部 GitLab MR 代码 Review。
- 内部 GitLab Merge Request 元信息和 diff 拉取，生成本次 Review 的不可变输入快照。
- 可插拔 code-review 执行器，一期默认对接 Claude Code `code-review` skill；未配置外部命令但已有可用 Chat 模型网关时，可通过模型网关适配器输出结构化 Review 报告。
- 产品详细设计、技术方案和代码 Review 报告的人工确认、修改后采纳、驳回重跑、要求补充信息和内部归档。
- LangGraph 运行内核，支持检查点、人工中断和恢复。
- PostgreSQL + pgvector 知识检索，支持权限过滤和引用来源。
- OpenAI-compatible 模型网关，统一聊天和 embedding 调用。
- 模拟 Issue 回写和 Markdown 方案导出。
- 知识沉淀候选生成、审核、批准和拒绝。
- 知识文档主动导入、索引状态查看、权限过滤检索、索引失败重试和归档。
- 产品、需求、AI 任务、知识中心和审计与运行必须有可用的独立列表、详情或等价入口。
- 首页、Bug 管理、研发运营看板和用户洞察/迭代规划在 MVP 可作为导航入口、真实空状态或禁用态展示；未接入真实数据源时不得返回示例数据、占位统计或伪造结果，不得暗示 v1.1/v1.2 能力已经完成。
- 审计事件记录与查询。
- Docker Compose 本地部署。

**v1.1 研发任务扩展**:
- AI 任务类型扩展到代码开发辅助和自动化测试，并增强 code-review 执行器的可替换能力。
- 开发计划和任务拆解的人工确认。
- Bug 管理支持 AI 自动测试发现 Bug 和人工测试登记 Bug，按产品、版本、模块、严重程度和状态跟踪。

**v1.2 研发运营扩展**:
- AI 任务类型扩展到发布上线评估和上线后分析。
- AI 助手升级为能力编排入口，支持 `@` 显式引用知识库、插件动作、AI Agent/Skill 和定时任务，在助理内生成可确认的 AI 能力配置、插件管理配置、定时作业配置和作业运行草案。
- 本地 GitLab 代码库每日提交采集、代码质量审核和人员代码提交情况统计，按产品归属汇总。
- 基于线上环境运行日志的产品运营状态监控分析，按产品、模块、环境和时间窗口展示错误率、延迟和核心业务事件。
- 实际业务系统用户使用数据采集，按产品、模块、功能、用户群体和时间窗口统计活跃、访问、转化、停留、异常退出和低使用功能。
- 用户使用反馈收集，支持满意度、问题反馈、改进建议、投诉、来源渠道、标签和关联产品模块。
- AI 迭代规划建议，结合产品规划、需求池、使用趋势、反馈热度、Bug、线上日志、发布记录和研发投入，输出下阶段优先迭代需求清单。
- Jenkins 发布数据采集，按产品和版本展示构建、部署、失败原因、发布频率和最近成功发布。
- 定时系统作业管理，支持管理员维护采集、聚合、看板刷新和 AI 分析任务的计划、启停、失败重试、运行台账和人工触发。
- AI 能力配置中心，支持管理员维护 Agent、Skill、任务类型默认配置和定时作业 AI 装配；普通业务用户只能选择已启用配置，不得修改系统提示词、工具权限或模型密钥。
- 首页 IT 团队看板，展示需求、研发进展、Bug、线上系统健康、核心业务运行情况、用户使用趋势、用户反馈趋势、AI 迭代规划建议、发布状态和知识沉淀统计。
- 软件研发全流程感知视图，按产品、版本、模块、需求和 AI 任务串联需求、设计、方案、提交、Review、测试、Bug、发布、线上日志、用户使用、用户反馈、迭代规划建议、知识沉淀和审计事件。

### Out of Scope

- 自动修改代码、自动提交 PR、自动部署上线。
- v1 MVP 不向 GitLab 回写 Review 评论、审批状态、request changes、合并状态或分支变更。
- 复杂可视化流程编排器。
- 深度企业 SSO、组织通讯录和跨租户商业化能力。
- 真实 GitHub/Jira/飞书等外部系统双向集成；v1 系列仅做模拟回写或预留适配器。
- 多模型自动竞价、复杂 RAG 平台化治理和大规模多租户隔离。

## 功能详情

### 功能点 1：研发需求到 PRD 草案

**描述**: 用户提交研发需求并审批通过后，系统通过 `product_detail_design` 任务生成正式产品详细设计，包括详细 PRD、交互说明、页面字段、业务规则和验收标准。审批前可展示轻量需求评估或 PRD 草案预览，但不作为独立 MVP 任务类型。

**规则**:
- 输入必须归属于一个业务大脑，v1 默认 `rd_brain`。
- PRD 草案进入人工确认点后才能继续生成技术方案。
- 信息不足时必须暂停并要求补充，而不是继续编造方案。

**异常处理**:
- 缺少必填字段时返回校验错误。
- 模型调用失败时任务进入失败或可重试状态，并记录审计事件。

### 功能点 2：研发全链路 AI 任务

**描述**: 在需求审批并排期后，AI 任务按业务阶段提供产品详细设计、技术方案、代码开发辅助、代码 Review、自动化测试、发布上线评估和上线后分析能力。v1 MVP 要求产品详细设计、技术方案和内部 GitLab MR 代码 Review 任务闭环；开发计划、自动化测试、发布上线评估和上线后分析为后续扩展能力。

**任务类型**:
| 任务类型 | 输入 | 输出 | 人工确认点 | 阶段 | 是否阻塞 MVP |
|----------|------|------|------------|------|--------------|
| `product_detail_design` | 已排期需求、产品上下文、历史知识、业务规则 | 详细 PRD、交互说明、页面字段、业务规则、验收标准 | 产品负责人确认详细设计 | v1 MVP | 是 |
| `technical_solution` | 产品详细设计、系统架构、代码仓库上下文、技术规范 | 技术方案、模块边界、接口设计、数据模型、风险依赖 | 研发负责人确认技术方案 | v1 MVP | 是 |
| `development_planning` | 技术方案、任务拆解、代码仓库上下文 | 开发任务清单、代码变更建议、实现步骤、待修改文件建议 | 研发负责人确认开发计划 | v1.1 | 否 |
| `code_review` | 内部 GitLab MR 元信息和 diff 快照、关联需求、技术方案、产品上下文、项目规范 | 结构化 Review 报告、问题清单、风险等级、文件/行号、修改建议、执行器元数据 | Reviewer 确认 Review 报告 | v1 MVP | 是 |
| `automated_testing` | PRD、验收标准、技术方案、已有测试用例 | 测试用例建议、自动化测试脚本建议、测试结果分析、Bug 登记建议 | 测试负责人确认测试结论 | v1.1 | 否 |
| `release_readiness` | 需求、代码质量、测试结果、Bug 状态、Jenkins 发布记录、线上日志历史 | 上线检查清单、发布风险评估、回滚建议、是否可发布结论 | 发布负责人确认上线 | v1.2 | 否 |
| `post_release_analysis` | 线上运行日志、核心业务事件、错误率、延迟、发布版本 | 上线后健康报告、异常趋势、疑似回归 Bug、优化建议 | 运营或研发负责人确认分析结论 | v1.2 | 否 |

**任务编排规则**:
| 任务类型 | 触发方式 | 前置依赖 | 阶段 |
|----------|----------|----------|---------|
| `product_detail_design` | 需求排期后自动或人工生成 | 已排期需求、产品上下文 | v1 MVP |
| `technical_solution` | 产品详细设计确认后生成 | 已确认产品详细设计 | v1 MVP |
| `development_planning` | 技术方案确认后生成 | 已确认技术方案 | v1.1 |
| `code_review` | 研发负责人或 Reviewer 选择内部 GitLab Merge Request 后创建 | 已确认技术方案、产品 GitLab 项目绑定、MR 元信息和 diff 快照 | v1 MVP |
| `automated_testing` | 人工上传测试结果或测试任务上下文后创建 | 验收标准、测试用例或测试结果 | v1.1 |
| `release_readiness` | 发布前由发布负责人创建 | 代码质量、测试结果、未关闭 Bug、发布记录 | v1.2 |
| `post_release_analysis` | 发布后由运营或研发负责人创建 | 发布版本、线上日志、核心业务事件 | v1.2 |

`product_detail_design`、`technical_solution` 和 `code_review` 构成 v1 MVP 的最小研发闭环；`code_review` 需要基于已确认技术方案和 GitLab MR / GitHub PR diff 快照单独创建，不要求由最初需求审批流一次性自动生成。其他任务类型是同一需求或产品上下文下的独立阶段任务，可在满足前置输入后单独创建。

**规则**:
- 所有任务类型都必须保留 `task_type`、产品上下文、来源需求或关联对象，并写入审计事件。
- 高影响 AI 产出必须进入人工确认点后才能继续下一阶段或进入回写。
- v1 MVP 代码 Review 任务必须通过产品绑定的 GitLab/GitHub 代码库读取 Merge Request / Pull Request 元信息和 diff，并保存不可变输入快照；GitHub 仓库应支持先列出可访问 PR，避免用户必须手工猜测 PR 编号。
- code-review 执行器必须是可插拔边界，一期默认对接 Claude Code `code-review` skill；本地联调可在外部命令缺失且模型网关可用时自动复用模型网关适配器；报告进入人工确认后才能归档为正式 Review 结论。
- v1 MVP 代码 Review 的 `writing_back` 仅表示 AI Brain 内部报告归档，不得回写 GitLab/GitHub 评论、审批状态、request changes、合并状态或分支变更。
- 代码开发辅助任务在 v1 系列中只生成方案、步骤、待修改文件建议和模拟 Issue，不自动修改代码、不自动提交 PR。
- 自动化测试任务可以登记来源为 `ai_auto_test` 的 Bug，但 Bug 进入修复前仍需可追踪的产品归属和复现信息。
- 发布上线评估任务只能给出风险判断和建议，v1 系列不自动部署上线。

**异常处理**:
- 人工拒绝可触发重跑、补充信息或取消。
- GitLab 项目未绑定、MR 不存在、权限不足、MR diff 过大、GitLab API 超时或限流时，任务进入 `failed` 或 `waiting_more_info`，并展示可操作错误原因。
- code-review 执行器失败时记录执行器类型、错误码、trace_id、失败阶段和是否可重试。
- 回写、Review、测试或发布评估失败可查询失败原因并重试。

### 功能点 3：知识检索与知识沉淀

**描述**: 工作流在生成内容前检索企业知识，并在任务完成后生成知识沉淀候选。

**规则**:
- 检索必须在数据库层执行权限过滤。
- AI 输出中引用知识时必须带来源引用。
- 知识沉淀初始状态为 `pending`，人工批准后才能进入正式知识库。

**异常处理**:
- 无相关知识时允许继续生成，但需明确标识上下文不足。
- Embedding 不可用时不得阻断知识入库；系统必须先保存文本 chunk 并进入 `text_indexed`，知识检索以关键词模式继续返回带来源的可访问结果，同时记录向量索引错误，待 Embedding 恢复后可补建向量索引。

### 功能点 4：审计与可追踪

**描述**: 所有写操作、高影响 AI 动作和人工确认都记录审计事件。

**规则**:
- API 响应必须返回 `trace_id`。
- 审计事件需关联用户、任务、运行和动作类型。
- 不默认存储完整 prompt 或模型输出，只记录元数据、摘要或哈希。

### 功能点 5：研发运营与产品洞察看板

**描述**: 平台按产品归属登记和查看 GitLab 代码提交与质量、Jenkins 发布和线上运行日志，并在首页 IT 团队看板和日志监控页面展示。

**规则**:
- GitLab 提交、代码质量审核、Jenkins 发布和线上日志都必须能映射到产品、版本或模块；日志监控页面不再提供采集运行记录和待归属数据队列入口。
- GitLab 指标至少包含提交量、活跃作者、Merge Request、变更量、质量评分和风险数量。
- Jenkins 指标至少包含 job、build_id、状态、部署环境、发布版本、触发人、耗时、失败原因和最近成功发布。
- 线上日志指标至少包含错误率、P95 延迟、核心业务事件数量、top errors 和异常趋势。
- 用户使用和用户反馈由用户洞察页面承载，不作为日志监控页面主功能。
- 首页看板默认展示需求、研发进展、Bug、线上系统健康、核心业务运行情况、用户使用趋势、用户反馈趋势、AI 迭代规划建议摘要、发布状态、代码质量和知识沉淀统计。
- 指标应支持按产品、模块、环境、用户群体和时间窗口筛选，并支持下钻到对应主体明细。

**异常处理**:
- 外部数据源采集失败时保留来源指标本身的失败状态或错误说明，不通过独立采集运行记录页面处理。

### 功能点 6：Bug 管理

**描述**: AI 自动测试和人工测试发现的问题统一进入 Bug 管理，支持登记、分派、修复、验证和关闭。

**规则**:
- Bug 来源包括 `ai_auto_test` 和 `manual_test`。
- Bug 必须归属于产品，可选关联版本、模块、需求、AI 任务、测试用例、GitLab 提交、Jenkins 发布或线上日志事件；人工登记 Bug 时目标版本可选择同产品未归档迭代版本，测试中或已发布版本仍可继续登记缺陷归属。
- Bug 状态包括 `open`、`triaged`、`needs_info`、`assigned`、`fixed`、`verified`、`closed`、`reopened`。
- 严重程度至少包括 `blocker`、`critical`、`major`、`minor`。
- 重复 Bug 通过 `duplicate_of_bug_id` 归并到主 Bug，不重复进入修复队列。

**异常处理**:
- AI 自动测试登记的 Bug 缺少复现信息时进入 `needs_info`，由测试负责人或测试人员补充复现步骤和证据。
- 重复 Bug 应关联到已有 Bug，不应重复进入修复队列。

### 功能点 7：软件研发全流程感知

**描述**: 平台围绕产品、版本、模块、需求和 AI 任务建立研发上下文图谱，把需求、产品详细设计、技术方案、代码提交、代码 Review、自动化测试、Bug、发布、线上日志、用户使用、用户反馈、迭代规划建议、知识沉淀和审计事件串联起来。

**规则**:
- 每个 AI 任务必须保留来源需求、产品上下文和 task_type；代码、测试、发布、线上日志、用户使用和用户反馈数据必须尽量映射到产品、版本、模块、需求或 AI 任务。
- v1 MVP 的全流程感知至少支持从需求查看下游产品详细设计任务、技术方案任务、人工确认、模拟 Issue、知识沉淀和审计事件。
- 当前实现已支持把已入库真实 Bug、GitLab 每日代码指标、Jenkins 发布记录、线上日志指标、用户使用指标、用户反馈和迭代规划建议纳入全流程感知链路；无法归属队列已具备真实记录、处理和审计入口；外部自动采集器和真实外部系统双向写回按后续增强推进。
- 全流程感知视图至少展示上游依据、当前阶段、下游影响、风险信号、关联负责人和下一步建议。
- 风险信号包括需求变更、设计未确认、代码质量下降、Review 高风险问题、测试失败、未关闭 blocker/critical Bug、发布失败、线上错误率或延迟异常、关键路径转化下降、核心功能低使用、负面反馈集中、需求价值与使用证据不匹配。
- 用户应能从需求详情、任务详情、Bug 详情、发布记录、用户反馈、迭代规划建议和首页看板互相跳转，保留产品、版本、模块、需求和时间范围上下文。

**异常处理**:
- 无法归属产品或需求的数据进入待归属队列，不参与主体级风险结论。
- 上下游链路缺失时明确标识缺口，不得把缺失上下文当作无风险。

### 功能点 8：用户使用洞察与 AI 迭代规划

**描述**: 平台采集实际业务系统用户使用数据和用户反馈，并结合产品规划、需求池、研发状态、Bug、线上日志和发布记录，主动生成下阶段优先迭代需求规划建议。

**规则**:
- 用户使用数据必须能映射到产品、模块、功能、用户群体和时间窗口。
- 用户反馈必须记录来源渠道、反馈类型、满意度或情绪倾向、标签、关联产品模块和处理状态；有价值反馈可由产品负责人、研发负责人或管理员转成正式需求，转需求时必须指定归属产品，可选归属版本和模块，反馈状态同步为 `linked` 并记录关联需求。
- 需求必须记录来源，至少覆盖业务部门、产品规划、用户反馈、内部调研和其他来源；用户反馈转需求生成的需求来源固定为用户反馈，迭代规划建议转需求生成的需求来源固定为产品规划。
- AI 迭代规划建议必须说明推荐理由、用户证据、业务价值、风险信号、依赖条件、预估研发投入和建议优先级。
- AI 只能生成需求规划建议，不能自动创建正式需求、变更产品路线图或调整迭代排期；正式采纳必须由产品负责人确认。
- 规划建议应能关联到已有需求、Bug、用户反馈、使用指标、线上日志或发布记录，避免无法追溯的泛化建议。
- 用户洞察采集、日志 AI 分析和迭代建议生成可以由定时系统作业触发；AI 生成类作业只能写入建议、摘要、风险信号或候选结果，不得自动创建正式需求、Bug、发布结论或变更产品路线图。
- 定时 AI 作业必须使用平台维护的 Agent/Skill 配置，并保存本次运行的 Agent、Skill、模型网关、Prompt 模板、输出 Schema、工具策略和上下文范围快照，便于审计和复盘。

**异常处理**:
- 使用数据不足或反馈样本过少时，规划建议必须标识证据不足，不得给出确定性排序。
- 无法归属产品或模块的使用数据和反馈进入待归属队列，不参与产品级优先级结论。

### 功能点 9：定时系统作业与 AI 能力配置

**描述**: 平台管理员可以把研发运营采集、用户洞察采集、线上日志 AI 分析、首页看板刷新、生命周期风险刷新和 AI 迭代建议生成配置为定时系统作业。需要 AI 参与的作业通过 Agent 和 Skill 装配执行能力，并统一走模型网关、工具白名单、配置快照和审计。

**规则**:
- 定时作业类型至少覆盖 `gitlab_daily_code_metric_collect`、`jenkins_release_collect`、`online_log_metric_collect`、`user_usage_metric_collect`、`user_feedback_collect`、`user_feedback_insight_extract`、`code_repository_inspection`、`online_log_ai_analysis`、`iteration_plan_suggestion_generate`、`dashboard_snapshot_refresh` 和 `lifecycle_context_refresh`。
- 作业执行模式分为 `deterministic`、`ai_assisted` 和 `ai_generated`：确定性作业可以直接写真实指标；AI 辅助作业可写分析摘要和风险信号；AI 生成作业只能写候选建议或待确认结果。
- `user_feedback_insight_extract`、`online_log_ai_analysis` 和 `iteration_plan_suggestion_generate` 属于平台 AI 作业；即使页面误选确定性执行，服务端也必须按 AI 生成链路校验 active Agent、Skill 和模型网关配置，避免运行时静默跳过大模型处理。其他作业一旦选择 `ai_assisted` 或 `ai_generated` 执行模式，前端保存前和服务端创建/修改时同样必须要求选择 AI 模型、Agent 和 Skill。
- 第一阶段 Agent 采用页面表单配置，维护执行角色、系统提示词、默认模型网关、默认 Skill、执行策略和工具策略；暂不要求 Agent 文件包上传。
- Skill 支持两种来源：页面表单配置和 zip 文件包上传。Skill 文件包必须包含 `skill.yaml` 或 `SKILL.md`，可携带 prompt、schema、示例和规则文档；服务端校验后保存到本地 Skill 存储目录，数据库只保存元数据、manifest、文件 URI、checksum 和启停状态。
- 定时作业可以指定 Agent、Skill、模型网关覆盖项和执行策略；AI 执行时必须从本地 Skill 文件包加载内容并冻结到运行快照，不得只引用可变文件路径。
- 每次运行必须生成运行实例，记录触发方式、计划时间、开始/结束时间、状态、导入数量、错误信息、collector run 关联和解析后的 AI 配置快照。
- 运行记录列表必须提供详情、复跑、复制配置和成功运行生成模板入口；复跑基于历史运行的 `scheduled_job_id` 重新触发当前作业配置，并在新运行实例和 `scheduled_job_run.succeeded/failed` 审计 payload 中记录触发方式 `manual_rerun` 与来源运行 `source_run_id`，普通手动触发记录为 `manual`，运行响应和运行列表还必须返回轻量 `source_run_summary`，用于在详情里对比来源状态、导入数、错误码和本次结果；审计 payload 还应包含状态、导入数量、collector run 和插件调用上下文，完成后自动打开新的运行详情，便于排查失败、验证修复和对比前后结果。运行记录页签顶部必须提供运行健康概览，聚合总运行数、成功率、失败率、平均耗时、AI 调用次数、Token 总量、插件调用次数、动作写入成功率、失败原因、最近失败和慢运行，用于快速判断取数、AI 处理和结果写入链路健康度。运行详情必须提供 Trace DAG，把数据连接、Runner 执行、Skill/AI 处理、结果写入和业务副作用节点的输入、输出、耗时、重试和错误定位可视化；成功运行生成模板时只生成新增作业草稿并保留运行来源。复制配置仅生成新增作业草稿，复制弹窗必须提示来源作业或运行快照；用户确认保存后在 `config_json.template_source` 和 `scheduled_job.created/updated` 审计 payload 中保留来源作业或来源运行快照，作业列表和运行详情必须可见该模板来源。
- 作业不得绕过现有业务服务直接写不可审计数据；所有写入仍需复用产品、模块、时间窗口、权限、归属和审计校验。
- 外部系统访问默认只读；Agent/Skill 不保存密钥，不直接接触 GitLab/GitHub/Jenkins/日志平台 token，只能引用服务端凭据引用或模型网关配置。
- 对 MaxCompute 等数据仓库场景，管理员优先在连接中配置 endpoint、Params、Headers 和认证；定时作业选择数据连接并设置时间窗口，Skill 用自然语言描述分析规则，结果动作负责声明 `write_target`、`insights_path`、`records_imported_path` 等写入规则。必要时可展开高级 JSON 直接编辑 MCP tool、SQL 模板和结果映射。
- 对仓库质量/安全/规范巡检场景，管理员可创建 `code_repository_inspection` 定时作业，通过插件连接调用 GitLab/GitHub/SonarQube/SAST 或自建扫描服务获取结构化 finding；确定性执行可直接使用扫描器结构化结果，`ai_assisted` 或 `ai_generated` 执行必须先调用模型网关，按 Agent/Skill 将扫描结果归一化为 `repository_id/branch/commit_sha/risk_level/summary/findings` 后再进入结果动作。作业支持多个 `result_actions`，默认先写入代码巡检报告，再按严重级别阈值自动创建 `source=code_inspection` Bug，并可登记邮件、钉钉机器人等消息通知反馈。巡检报告详情必须展示来源作业、来源运行、数据连接、结果动作和插件调用，来源作业和来源运行可跳转到任务中心 / 定时作业，其中来源运行链接直接打开运行详情，便于从报告反查配置和插件执行上下文。巡检结果必须保留提交人姓名、邮箱或用户名维度，支持按提交人筛选；扫描器 severity 可映射到平台统一级别；同一仓库、规则、文件、行号和提交人的开放问题不得重复创建 Bug。运营治理 / 代码巡检页面必须提供治理概览，基于当前筛选聚合报告数、问题数、严重问题、Bug 覆盖率、规则维度统计、仓库/分支/提交人风险排行、最近趋势和严重问题 SLA，帮助管理者判断巡检闭环是否持续收敛。
- 插件动作的结果写入目标必须由服务端注册表统一提供，首批包括 `scheduled_job_result`、`user_feedback_insights`、`code_inspection_reports` 和 `email_notifications`；注册表必须定义目标中文标签、表单标签、默认 `result_mapping` 和可视化 JSONPath 字段，前端按注册表渲染字段并仅在服务端不可用时使用兼容兜底。选择代码巡检报告时，页面展示仓库、分支、提交 SHA、风险级别、摘要和 finding 列表 JSONPath 字段；选择邮件通知记录时，页面展示收件人、主题、投递状态和消息 ID JSONPath 字段。结果写入记录属于定时作业运行产物，必须在“任务中心 / 定时作业 / 运行记录 / 运行结果详情”中按单次 `scheduled_job_run_id` 展示；详情中在数据连接、Skill/AI 处理、结果动作反馈之后展示最终写入记录，邮件通知记录展示主题、投递状态、消息 ID 和收件人摘要，用户洞察、代码巡检报告和未来新增写入目标也必须保留通用预览和反馈 JSON。插件管理只负责插件、连接、动作配置和调用日志，不展示定时作业运行结果记录。
- 代码巡检运行时必须应用动作或作业输出 `result_mapping`，把三方插件响应中的 `repository_id/branch/commit_sha/risk_level/summary/findings` 映射为平台统一结构；JSONPath 支持 `$` 根节点以兼容 GitHub/GitLab 等接口返回根数组的场景。
- 插件动作试运行必须在请求预览、响应摘要和 JSONPath 命中之外展示写入预览，包括写入目标、预计写入数量、候选记录数量、预览值、报告字段预览或样例记录，便于管理员在保存定时作业前确认动作结果是否可写入目标表；每次试运行必须写入 `plugin_action.trial_succeeded/failed` 审计事件，记录插件、动作、连接、连接环境、写入目标、状态、耗时和错误码等轻量上下文，不保存完整请求响应、输入 payload 或密钥。
- GitLab、GitHub、邮箱和 AI 执行器作为官方标准插件内置提供，用户不能修改或删除插件定义；插件管理必须提供官方插件市场，用于只读展示官方插件发布方、简介、推荐场景、连接模板版本、动作模板、安装状态、已配置连接数和动作数，并支持从市场项直接进入新增连接表单或新增动作表单。管理员只在连接里配置实例 endpoint、认证引用和平台参数，连接列表必须支持按 `default/dev/test/staging/prod/sandbox` 环境筛选，便于在多环境实例间切换维护；定时作业新增/编辑时也必须提供连接环境筛选，仅用于过滤数据连接选项，不写入作业配置，避免生产、测试、沙箱连接混选。GitLab/GitHub/邮箱/AI 执行器连接默认参数必须由服务端插件市场 `connection_defaults` 返回，页面新增连接和 AI 助手连接草案复用同一模板；GitLab 连接默认提供 `PRIVATE-TOKEN` Header 认证、`api_version/group_id/project_id` Params；GitHub 连接默认提供 Bearer Token、`Accept` 和 `X-GitHub-Api-Version` Headers，以及 `owner/repo` Params；邮箱连接默认提供邮件网关 API endpoint、`Authorization` Header 认证、`Content-Type=application/json` Header，以及 `send_protocol/receive_protocol/smtp_host/smtp_port/imap_host/imap_port/pop3_host/pop3_port/mailbox_folder/poll_since/default_from/default_to/subject_template` 等收发参数；AI 执行器连接默认使用 `runner_polling`，通过 `executor_type` 在 `codex/claude/hermes/openclaw` 中选择执行器，并配置 `runner_id/workspace_root/instruction_timeout_seconds/result_callback_url` 等受控 Runner 参数。插件动作新增时必须从服务端动作模板目录动态获取 MaxCompute 每周用户反馈、GitHub 代码巡检、GitLab 代码巡检、邮箱通知发送、邮件收取、AI 执行器下达指令和执行器结果同步等场景模板，并自动填充官方插件、动作编码、HTTP/MCP/Runner 请求配置、默认 Params/Headers、表单默认值和对应结果映射；前端和 AI 助手不得在服务端模板缺失时硬编码生成这些官方动作，只能提示模板缺失或要求刷新服务端模板目录；邮箱通知模板默认使用 `email_notifications` 写入目标，便于试运行和作业详情展示投递反馈；AI 执行器真实运行必须经过隔离 Runner、安全授权和审计，不得在普通 HTTP 插件中裸执行本机命令；Runner 通过心跳、任务认领、日志追加和完成回写同步本地 Codex/Claude/Hermes/OpenClaw 执行结果，定时作业在任务排队或执行中保持 `running`，完成回写后才进入成功或失败终态；管理员必须能轮换 Runner Token、查看任务日志、取消任务并通过超时熔断处理失联执行；从插件市场点击“创建动作”必须直接打开对应模板，用户可继续用可视化字段或高级 JSON 调整。
- 插件连接测试必须展示请求调试台，包含最终请求 URL、Params、Headers、Header 来源、原始请求配置、系统动态变量解析明细、完整请求 JSON 和远端响应摘要；当 Params/Headers 中存在 `{{current_date-7}}` 等变量时，页面必须展示变量位置、原始表达式、偏移天数、解析值、最终请求值和解析时区，便于排查日期分区和 Header 配置问题；当当前连接未配置动态变量时，也必须显示“未检测到动态变量”空状态，避免用户误判调试台未生效。每次连接测试完成后，连接列表必须保留并展示最近一次测试的状态、耗时、错误码或失败步骤，便于关闭诊断弹窗后继续排障；诊断弹窗必须进一步提供请求回放台，展示最近 N 次测试记录、变量解析前/解析后差异、失败修复建议，并可将当次请求复制为动作模板草案，减少把可用连接改造成结果动作时的重复配置。运行结果详情必须优先展示“数据连接获取内容、经过 Skill 处理后的内容、结果动作反馈内容”三段执行链路，再提供原始 JSON 作为高级排查入口。
- Skill 包默认只允许文本、Markdown、YAML 和 JSON 配置文件；脚本目录即使上传也不得在第一阶段自动执行，后续如启用脚本必须先引入沙箱和权限审批。
- 高风险 AI 输出必须进入已有确认流或候选表，例如迭代建议继续由产品负责人通过迭代规划确认流采纳、修改后采纳或驳回。

**异常处理**:
- 计划触发失败、锁竞争失败或配置停用时，运行实例应标记 `skipped` 或 `failed` 并记录原因，不生成业务数据。
- 模型网关失败、Skill 输出 Schema 校验失败或外部源超时时，作业进入可重试失败状态；重试必须复用原作业配置并生成新的运行实例。
- 无法自动归属产品、模块、需求或主体的数据进入待归属队列；低置信度 AI 结论不得进入正式看板风险结论。

### 关键交互

1. 任务详情页展示需求、任务类型、AI 中间结果、检索证据、GitLab MR / GitHub PR diff 快照、代码/测试/发布相关上下文、风险、Review 报告和审计轨迹。
2. 待确认弹窗支持采纳、修改后采纳、要求补充信息、拒绝重跑和取消。
3. 任务状态必须清晰展示 `draft`、`running`、`waiting_more_info`、`waiting_review`、`writing_back`、`completed`、`failed`。
4. 产品管理入口必须能维护产品、模块和 Git 资源；需求交付下的迭代版本入口必须能维护产品版本，并清晰标识规划中、开发中、测试中和已发布状态，支持直接查看当前版本需求清单，测试中、已发布和历史归档版本不可用于需求排期或新开发任务；Git 凭据只允许在新增/编辑时提交，列表和详情不得回显凭据引用或 token 明文。
5. 代码 Review 任务创建时必须选择已绑定且有权限的 GitLab Merge Request 或 GitHub Pull Request；任务详情展示 MR/PR 元信息、diff 快照、执行器信息、结构化 Review 报告、人工确认状态和内部归档结果。
6. 需求详情页必须展示原始需求、审批状态、生成的任务引用和关闭状态，不依赖任务详情才能理解需求。
7. 知识中心必须展示文档索引状态、索引失败原因、权限范围、检索结果来源和沉淀审核状态，并对 `index_failed` 文档提供重试入口。
8. 首页 IT 团队看板必须支持按产品下钻到需求、研发进展、Bug、发布、代码质量、线上运行指标、用户使用趋势、用户反馈趋势和 AI 迭代规划建议。
9. 日志监控必须展示 GitLab 提交与代码质量、Jenkins 发布记录和线上日志指标；页面不得再展示采集运行记录或待归属数据队列。
10. Bug 管理支持 AI 自动测试登记和人工测试登记，Bug 详情展示来源、复现信息、关联需求/任务/测试用例/提交/发布/日志、处理人和验证记录。
11. 软件研发全流程感知视图支持从需求下钻到设计、方案、代码提交、Review、测试、Bug、发布、线上日志、用户使用、用户反馈、AI 迭代规划建议、知识沉淀和审计事件。
12. AI 助手页面支持发送自然语言问题，展示助手回答、模型、延迟、建议追问、最近对话和历史消息；聊天记录按登录用户保存和隔离，模型调用日志不得保存完整用户问题、系统上下文、助手回答或 API Key。
13. 所有涉及日期或时间登记的前端弹窗必须使用 Ant Design DatePicker；日期字段按 `YYYY-MM-DD` 提交，时间窗口、开始时间和部署时间按 `YYYY-MM-DDTHH:mm:ssZ` 提交，不得退回普通文本输入。

## 验收标准

- [ ] AC1: 用户可以创建研发大脑 AI 任务，并通过 API 或页面启动工作流。
- [ ] AC2: 信息不足时任务暂停在补充信息状态，补充信息提交后回到 `draft`，再次启动后继续运行。
- [ ] AC3a: v1 MVP 中产品详细设计、技术方案和内部 GitLab MR 代码 Review 报告等高影响 AI 产出必须经过人工确认后才能进入下一阶段或归档。
- [ ] AC3b: v1.1 中开发计划和自动化测试结论必须经过人工确认后才能进入下一阶段或回写。
- [ ] AC3c: v1.2 中发布上线评估和上线后分析必须经过人工确认后才能进入发布建议采纳、风险处理或知识沉淀流程。
- [ ] AC4a: MVP-A 任务确认后可以导出 Markdown 方案。
- [ ] AC4b: MVP-C 任务完成后生成模拟 Issue，且重复触发不会产生重复结果。
- [ ] AC4c: MVP-C 任务完成后生成知识沉淀候选，候选必须经人工审核后才能进入正式知识库。
- [ ] AC5: 知识检索结果带来源引用，并在查询层完成权限过滤。
- [ ] AC6: 所有写操作和 AI 关键动作可在审计事件中追踪。
- [ ] AC7: Docker Compose 可以启动 web、api、PostgreSQL + pgvector 和 Redis，本地健康检查通过。
- [ ] AC8: 管理员可以维护产品、模块、Git 资源和需求交付下的迭代版本配置；迭代版本页可以维护版本级代码分支配置，同一版本可关联多个代码库并记录基准分支、开发分支、分支状态和创建来源；提交需求时必须选择有效产品，目标迭代版本可留空；需求管理可以按迭代版本查询需求；产品负责人或研发负责人可以将多条同产品需求从需求管理或迭代版本页批量归集到规划中或开发中的迭代版本；迭代版本页在版本进入测试中后仍可直接查看当前版本需求清单；版本推进到开发中、测试中和已发布时必须预览影响并同步符合条件的需求状态，其中开发中推进测试中必须把版本内已进入交付链路的需求统一同步为测试中；只有排期后的需求才能生成 AI 任务。
- [ ] AC9: 管理员可以维护平台默认模型网关配置，密钥不在 API 响应中明文返回。
- [ ] AC10: 产品、需求、AI 任务、知识中心和审计与运行必须有可用的独立列表、详情或等价入口；Bug 管理必须使用真实接口或真实空状态，并能展示、查询 Bug 归属迭代版本；日志监控和用户洞察/迭代规划在 v1 MVP 必须使用真实接口或真实空状态，完整维护能力按 v1.1/v1.2 验收。
- [ ] AC11: 需求审批和 AI 任务执行解耦；需求保留审批状态和原始业务事实，任务保留生成时的需求快照和产品上下文。
- [ ] AC12: 知识中心支持独立导入、索引状态查看、权限过滤检索、索引失败处理和沉淀审核。
- [ ] AC13: 审计事件可按任务、主体、用户、事件类型和时间范围追踪关键写操作和 AI 高影响动作。
- [ ] AC14: GitLab 每日提交和代码质量审核结果可以按产品、仓库和人员统计；无法归属产品的数据不得混入产品级统计。
- [ ] AC15: 线上运行日志可以通过真实登记、导入或自动采集生成产品运营状态监控分析，包括错误率、接口延迟、核心业务事件、top errors 和异常趋势。
- [ ] AC16: Jenkins 发布数据可以按产品和版本展示构建、部署、失败原因、发布频率和最近成功发布。
- [ ] AC17: 首页 IT 团队看板可以展示需求、研发进展、Bug、线上系统健康、核心业务运行情况、用户使用趋势、用户反馈趋势、AI 迭代规划建议摘要、发布状态、代码质量和知识沉淀统计，并支持下钻。
- [ ] AC18: Bug 管理支持 AI 自动测试和人工测试登记两类来源，并完成分派、修复、验证、关闭和重复归并闭环。
- [ ] AC19: 软件研发全流程感知视图可以按产品、版本、模块、需求和 AI 任务串联需求、设计、方案、代码提交、Review、测试、Bug、发布、线上日志、用户使用、用户反馈、迭代规划建议、知识沉淀和审计事件，并标识上下游缺口和风险信号。
- [ ] AC20: 用户使用洞察与 AI 迭代规划可以按产品、模块、功能、用户群体和时间窗口汇总使用数据与用户反馈，并结合产品规划、需求池、Bug、线上日志、发布记录和研发投入生成可追溯的下阶段优先迭代需求建议；无法归属的使用数据和反馈不得混入产品级建议；AI 建议必须经产品负责人确认后才能转为正式需求或进入迭代计划。
- [ ] AC21a: MVP-A 支持配置内部 GitLab 只读仓库绑定，基于授权产品 Git 资源预览 Merge Request 并生成不可变 MR diff 快照，快照作为后续 code_review 任务输入，不得静默覆盖。
- [ ] AC21b: MVP-B 支持基于内部 GitLab Merge Request diff 快照调用可插拔 code-review 执行器生成结构化 Review 报告，并在人工确认后仅归档到 AI Brain 内部；系统不得向 GitLab 回写评论、审批状态、request changes、合并状态或分支变更。
- [ ] AC22: AI 助手可以基于服务端脱敏系统上下文回答 AI Brain 系统配置、产品、需求、AI 任务、Git 仓库、模型网关状态和项目开发进展问题；聊天历史按当前登录用户保存和查询，跨用户不可读取；模型日志只记录脱敏元数据，不保存完整问题、回答或密钥。
- [ ] AC23: AI 助手支持通过 `@` 显式引用知识库、业务对象、插件、AI Agent/Skill、模型网关配置、定时作业和运行实例；助理可生成 AI 能力配置、插件管理配置、定时作业配置和手动运行草案，但所有写配置、外部调用、删除/停用和作业运行必须经有权限用户确认后才执行，并写入审计。
- [ ] AC24: 运营治理支持创建代码仓库巡检定时作业，定期扫描仓库质量、安全和规范问题；运行结果必须写入代码巡检报告表并可在运营治理 / 代码巡检页面查看详情，报告列表和详情必须能追踪来源作业、运行、数据连接、结果动作和插件调用；列表和详情支持提交人维度展示与筛选；严重问题可按阈值自动创建 `code_inspection` 来源 Bug 且同一开放 finding 不重复建 Bug；消息通知动作需记录邮件、钉钉机器人等反馈；代码巡检页面必须展示趋势、规则、仓库/分支/提交人排行和严重问题 Bug 覆盖 SLA；非全局用户只能读取授权产品范围内报告。

### MVP 演示验收路径

v1 MVP 验收应优先按以下演示脚本验证，避免只按单点 AC 判断局部完成：

1. 管理员登录，创建启用产品、版本、模块和内部 GitLab 只读仓库绑定。
2. 管理员确认默认模型网关已配置，密钥在页面和 API 响应中只显示掩码或 configured 标记。
3. 产品负责人创建研发需求并提交审批。
4. 产品负责人审批需求，系统生成 `product_detail_design` 任务并固化需求快照和产品上下文。
5. 产品负责人启动任务并确认产品详细设计输出。
6. 研发负责人基于已确认产品详细设计生成并确认 `technical_solution` 任务。
7. Reviewer 选择已绑定内部 GitLab 项目中的 MR，完成 MR 预览和 diff 快照。
8. Reviewer 创建并启动 `code_review` 任务，系统调用可插拔执行器生成结构化 Review 报告。
9. Reviewer 对 Review 报告执行确认或修改后采纳，系统只归档到 AI Brain 内部，不回写 GitLab。
10. 知识维护者查看知识沉淀候选并批准或拒绝。
11. 用户导出 Markdown 方案，查看模拟 Issue、内部 Review 报告归档和任务详情。
12. 管理员或授权用户在审计与运行中按需求、任务、Review 报告和知识沉淀查询关键审计事件。

验证过程中若 Bug 管理、研发运营看板、用户洞察/迭代规划或完整首页看板尚未交付，必须展示清晰的空状态或禁用态，不得返回示例数据或暗示 v1.1/v1.2 能力已经完成。

## 非功能需求

| 类型 | 要求 |
|------|------|
| 性能 | 常规列表和详情接口目标 P95 < 500ms；AI 工作流允许异步长任务。 |
| 安全 | 本地账号 + Bearer Token；写操作鉴权；知识检索数据库层权限过滤。 |
| 可用性 | AI 任务失败可查看错误、重试或取消；人工中断后可恢复。 |
| 可观测性 | 每个 API 响应返回 `trace_id`；关键动作写入审计事件；GitLab、Jenkins、线上日志、用户使用和用户反馈采集记录最后成功时间、失败原因和归属状态。 |
| 可维护性 | v1 MVP 保持模块化单体，避免过早拆分微服务。 |

## 上线计划

### 阶段一：MVP-A 基础 + GitLab 输入闭环
- 时间: v1 MVP 第一开发切片。
- 范围: FastAPI `/health`、React 入口、Docker Compose、PostgreSQL + pgvector、Redis、本地账号、基础产品配置、内部 GitLab 只读仓库绑定、MR 预览、MR diff 快照、需求创建与审批、产品详细设计任务、技术方案任务、人工确认、Markdown 导出和基础审计。
- 阻塞验收: AC1、AC2、AC3a 中产品详细设计和技术方案部分、AC4a、AC7、AC8、AC11、AC21a。

### 阶段二：MVP-B GitLab Review 闭环
- 时间: v1 MVP 第二开发切片。
- 范围: `code_review` 任务、可插拔 code-review 执行器、结构化 Review 报告、Review 报告人工确认、内部报告归档、不回写 GitLab、执行器错误语义、diff 超限处理和相关审计。
- 阻塞验收: AC3a 中 code_review 部分、AC6、AC13、AC21b。

### 阶段三：MVP-C 知识与治理闭环
- 时间: v1 MVP 第三开发切片。
- 范围: 知识文档导入、索引状态、权限过滤检索、知识沉淀候选审核、模拟 Issue、MVP 真实空状态入口和主体级审计体验增强。
- 阻塞验收: AC4b、AC4c、AC5、AC10、AC12。

### 阶段四：v1.1 研发任务扩展
- 时间: v1.1 完整演示增强。
- 范围: 开发计划、自动化测试任务、AI 自动测试 Bug 建议、code-review 执行器扩展能力和运行审计体验增强。
- 阻塞验收: AC3b、研发任务扩展相关验收；AC18 在 v1.1 验收 AI 自动测试 Bug 建议和人工登记基础流转，在 v1.2 验收完整 Bug 管理闭环。

### 阶段五：v1.2 研发运营扩展
- 时间: v1.2 研发运营扩展。
- 范围: GitLab 每日提交和代码质量指标采集、Jenkins 发布、线上运行日志、用户使用数据、用户反馈、AI 迭代规划建议、Bug 管理闭环、研发运营看板、首页 IT 团队看板下钻和全流程感知扩展。
- 阻塞验收: AC3c、AC14、AC15、AC16、AC17、AC18、AC19、AC20。
