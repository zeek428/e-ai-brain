# 测试策略

## 测试原则

1. **测试左移**: 尽早测试，开发阶段完成单元测试
2. **测试金字塔**: 单元测试 > 集成测试 > E2E 测试
3. **自动化优先**: 可自动化的测试必须自动化
4. **覆盖率要求**: 核心代码覆盖率 ≥ 80%

## 测试类型

### 单元测试

**目的**: 验证最小代码单元的正确性

**范围**:
- 函数、方法
- 类、模块
- 工具函数

**要求**:
- 覆盖率 ≥ 80%
- 隔离依赖，使用 Mock
- 执行时间 < 10s

### 集成测试

**目的**: 验证模块间交互的正确性

**范围**:
- API 接口
- 数据库操作
- 外部服务调用

**要求**:
- 使用测试数据库
- 测试完成后清理数据
- 执行时间 < 5min

### E2E 测试

**目的**: 验证完整业务流程

**范围**:
- 核心业务流程
- 关键用户路径
- 跨系统集成

**要求**:
- 使用独立测试环境
- 定期执行（每日/每周）
- 执行时间 < 30min

## 测试环境

| 环境 | 用途 | 数据 |
|------|------|------|
| Local | 本地开发测试 | Mock 数据 |
| Test | CI 自动化测试 | 测试数据 |
| Staging | 预发布验证 | 脱敏数据 |
| Prod | 生产环境 | 真实数据 |

## 测试工具

| 类型 | 工具 | 用途 |
|------|------|------|
| 后端单元测试 | Pytest / pytest-cov | FastAPI 领域模块、状态机、schema 校验和覆盖率。 |
| 前端单元测试 | Vitest / React Testing Library | React + TypeScript 组件、表单、状态展示和权限分支。 |
| 集成测试 | Pytest + 测试数据库 / TestContainers | API、PostgreSQL、pgvector、Redis、GitLab/Jenkins/模型网关适配器。 |
| E2E 测试 | Playwright | 端到端业务流程和浏览器交互验证。 |
| 性能测试 | k6 / JMeter | API、知识检索、看板和全流程感知查询压测。 |
| 覆盖率 | pytest-cov / Vitest coverage | 后端和前端覆盖率统计。 |

## 测试流程

```
编写代码 → 编写单元测试 → 本地测试 → 提交代码 → CI 自动测试 → 测试通过? → 合并代码
```

AI Brain 项目实现阶段的默认验证命令以实际工程脚本为准；后端优先使用 `uv run pytest`，前端优先使用项目锁定的 package manager 运行 Vitest 和 Playwright，Docker Compose 验证按部署 runbook 执行。

## 测试命名规范

```
test_<模块>_<功能>_<场景>

示例:
test_user_create_success
test_user_create_with_invalid_email
test_user_delete_not_found
```

## Mock 规范

### 何时使用 Mock
- 外部服务调用
- 数据库操作（单元测试）
- 时间依赖
- 随机结果

## 测试用例编写规范

### 用例编号规则
```
TC-{项目}-{模块}-{类型}-{序号}

项目级文档应优先使用全局唯一编号；若历史文档使用模块内序号，必须在文档中明确说明，避免自动化报告和验收映射混淆。

类型:
- FUNC: 功能测试
- BOUND: 边界测试
- ERR: 异常测试
- API: 接口测试
- PERF: 性能测试

示例:
TC-AUTH-LOGIN-FUNC-001  // 用户登录功能测试第1个用例
TC-AIBRAIN-TASK-ERR-001  // AI 任务异常测试第1个用例
```

### 用例优先级
| 优先级 | 说明 | 通过要求 |
|--------|------|----------|
| P0 | 核心功能，阻塞发布 | 必须 100% 通过 |
| P1 | 重要功能，影响主流程 | 必须 100% 通过 |
| P2 | 一般功能，影响体验 | ≥ 95% 通过 |
| P3 | 边缘场景 | ≥ 90% 通过 |

### 用例模板

```markdown
### TC-XXX-FUNC-001: {用例名称}

| 项目 | 内容 |
|------|------|
| 用例编号 | TC-XXX-FUNC-001 |
| 用例名称 | |
| 优先级 | P0 |
| 前置条件 | 1. <br>2. |

**测试步骤**:
| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | | |
| 2 | | |

**状态**: 待测试 / 通过 / 失败 / 阻塞
```

### 用例设计原则

1. **单一职责**: 一个用例只验证一个功能点
2. **可重复**: 用例可重复执行，结果一致
3. **独立性**: 用例之间相互独立，无依赖
4. **可追溯**: 用例关联需求，可追溯

### 测试类型覆盖要求

| 类型 | 覆盖内容 | 示例 |
|------|----------|------|
| 正向测试 | 正常业务流程 | 正确输入得到正确输出 |
| 边界测试 | 边界值、临界值 | 最大值、最小值、空值 |
| 异常测试 | 异常输入、异常状态 | 参数缺失、格式错误 |
| 兼容测试 | 多环境、多版本 | 浏览器兼容、版本兼容 |
| 性能测试 | 性能指标 | 响应时间、并发数 |

### 用例评审清单

- [ ] 用例编号符合规范
- [ ] 前置条件明确
- [ ] 测试步骤清晰可执行
- [ ] 预期结果明确可验证
- [ ] 覆盖正向和异常场景
- [ ] 关联需求已标注
