# 安全规范

## 安全原则

1. **最小权限**: 用户、任务、知识文档和模型配置都只暴露必要权限。
2. **默认鉴权**: 除 `/health` 和 `/api/auth/login` 外，API 默认需要 Bearer Token。
3. **权限下推**: 知识检索必须在数据库查询层过滤权限，不能只靠前端隐藏。
4. **AI 数据最小化**: 发往模型的数据需要脱敏、限长，并避免发送无关敏感信息。
5. **可审计**: 所有写操作和高影响 AI 动作必须写入审计事件。

## 认证与授权

### 认证方式

v1 使用本地账号 + Bearer Token。

| 接口 | 鉴权要求 |
|------|----------|
| `GET /health` | 不需要鉴权。 |
| `POST /api/auth/login` | 不需要 Bearer Token，但需要校验账号密码。 |
| `/api/*` 其他接口 | 默认需要 Bearer Token。 |

### 用户角色

| 系统角色 | 对应业务角色 | 权限摘要 |
|------|----------|----------|
| admin | 平台管理员 | 管理用户、业务大脑配置和平台设置。 |
| rd_owner | 产品负责人、研发负责人、发布负责人等高影响动作负责人 | 提交和审批需求，确认产品详细设计、技术方案、开发计划、代码 Review、自动化测试、发布评估和上线后分析等高影响 AI 产出。 |
| tester | 测试负责人、测试人员 | 查看任务和测试相关输出，登记、分派、验证或关闭授权范围内的 Bug。 |
| knowledge_owner | 文档/知识维护者 | 审核、修改后采纳或驳回知识沉淀，维护授权范围内的知识文档。 |
| viewer | IT 管理者、只读观察者 | 只读访问授权资源、看板和审计摘要。 |
| member | 普通项目成员 | 查看参与任务，创建需求，补充信息。 |

### Token 管理

- `ACCESS_TOKEN_EXPIRE_SECONDS` 控制访问令牌有效期。
- `APP_SECRET_KEY` 必须来自环境变量，不能提交真实值。
- Token 不写入日志，不出现在错误响应中。

## 密钥管理

### 环境变量

| 变量 | 用途 | 要求 |
|------|------|------|
| APP_SECRET_KEY | 应用签名密钥 | 必填，真实值只放 `.env` 或密钥管理系统。 |
| DATABASE_URL | PostgreSQL 连接 | 不提交真实密码。 |
| REDIS_URL | Redis 连接 | 不提交真实密码。 |
| MODEL_GATEWAY_API_KEY | 模型供应商密钥 | 不入库、不进日志。 |
| MODEL_GATEWAY_BASE_URL | 模型网关地址 | 可在 `.env.example` 中提供占位。 |

后续生产环境应接入 Vault 或云厂商密钥管理。

## 输入验证

### 必须验证的输入

- URL 路径参数。
- Query 参数。
- Request Body。
- Header 中的认证和追踪字段。
- Markdown 文档上传内容。
- 模型网关返回的结构化 JSON。

### 防护措施

| 风险 | 防护措施 |
|------|----------|
| SQL 注入 | 使用参数化查询或 ORM 参数绑定。 |
| XSS | 前端渲染 Markdown/AI 输出时做 HTML 清理和输出编码。 |
| 路径遍历 | 文档导入和导出文件名必须规范化。 |
| 越权访问 | API 层校验任务参与关系、角色和知识文档权限。 |
| 并发覆盖 | human_reviews 使用 version 乐观锁。 |

## AI 数据安全

### 发送模型前

- 过滤 API Key、Token、密码、连接串。
- 对手机号、邮箱等敏感信息做脱敏。
- 限制单次输入长度。
- 只发送完成当前节点所需的最小上下文。

### 模型调用日志

允许记录：

- provider
- model
- purpose
- token 使用量
- latency_ms
- status
- error code/message 摘要

默认禁止记录：

- 完整 prompt
- 完整模型输出
- API Key 或 Bearer Token
- 未脱敏个人信息

## 知识权限

检索时必须根据用户角色、业务大脑、文档权限和产品/版本上下文过滤。

不能出现：

- 后端返回无权限 chunk，前端再隐藏。
- AI 引用用户无权访问的文档。
- 审计记录缺失导致无法追踪知识来源。

## 审计要求

以下动作必须写入审计事件：

- 登录成功和失败。
- 创建、启动、重试、取消 AI 任务。
- 内部 GitLab MR 预览、diff 快照创建和快照复用。
- code-review 执行器开始、失败、超时、schema 校验失败和报告生成。
- 代码 Review 报告人工确认、修改后采纳和内部归档。
- 人工确认、修改后采纳、拒绝、要求补充信息。
- 知识文档导入、索引、删除或权限变更。
- 知识沉淀批准、修改后批准、拒绝。
- 模拟 Issue 回写。
- 用户使用数据、用户反馈、Jenkins、线上日志和 GitLab 指标采集失败或无法归属。
- 模型调用失败和 Graph 运行失败。

审计事件字段以项目级技术规格的审计事件 schema 为准，至少包含：`id`、`ai_task_id`、`event_type`、`actor_type`、`actor_id`、`event`、`created_at`。API 响应 envelope 的 `trace_id` 用于请求追踪，不要求写入当前审计事件表。

## 安全清单

### 开发阶段

- [ ] 新 API 默认鉴权，明确列出匿名接口例外。
- [ ] 所有外部输入有 schema 校验。
- [ ] 数据库查询使用参数化方式。
- [ ] 知识检索在查询层过滤权限。
- [ ] AI 调用前执行敏感信息过滤。
- [ ] MR diff、用户反馈和使用数据进入模型前已脱敏、限长并只保留当前任务必需上下文。
- [ ] GitLab token、模型 API Key、Bearer Token 和用户级明细不得传给 code-review 执行器。
- [ ] 日志不包含密钥、Token、完整 prompt。

### 测试阶段

- [ ] 覆盖未登录、无权限、过期 review version、无权限知识检索。
- [ ] 覆盖 GitLab 项目未绑定、MR 不存在、权限不足、diff 过大和不回写 GitLab。
- [ ] 覆盖 code-review 执行器失败、超时和结构化报告 schema 校验失败。
- [ ] 覆盖模型网关失败和结构化 JSON 校验失败。
- [ ] 覆盖重复回写和 MR 快照复用幂等性。
- [ ] 覆盖审计事件写入。

### 上线阶段

- [ ] `.env` 未提交真实密钥。
- [ ] `APP_SECRET_KEY`、模型 API Key 和 GitLab 只读凭据已配置。
- [ ] `/health` 可用但不泄露敏感配置。
- [ ] 数据库和 Redis 不对公网暴露。
- [ ] code_review 任务验证不会向 GitLab 回写评论、审批状态、request changes、合并状态或分支。
- [ ] 备份和恢复策略已确认。
