# AI Brain 功能迭代需求驱动研发协同演示设计

## 状态与目的

- 状态：待用户审阅
- 日期：2026-07-21
- 目标：在真实本地 AI Brain 环境中，以 AI Brain 自身的一项功能迭代需求完整演示需求评估、版本协同、AI 开发测试、真人审核、远程提交、知识沉淀和岗位经验复用。

本设计只定义演示案例和执行边界，不改变现有定时作业，也不触发部署。

## 演示需求

**统一研发执行策略支持维护岗位经验复用的可信域配置，并在研发协同工作项中展示已批准经验的引用与证据。**

该需求解决两个已确认的真实可观察性缺口：

1. 后端已经按策略快照的仓库/工具可信域、风险、岗位、产品和容量约束检索批准经验，但策略编辑页面没有可信域配置入口。
2. 规划服务会将批准经验写入工作项 `input_contract.approved_role_experience_context`，但工作项 DAG 页面没有展示该引用，研发人员无法在页面确认复用发生。

## 范围与保护边界

| 项目 | 约束 |
| --- | --- |
| 产品 | 已有 `AI Brain` 产品，不创建替代演示产品 |
| 代码分支 | `codex/rd-collaboration-v2`，远程为 `origin/codex/rd-collaboration-v2` |
| 迭代版本 | 评估通过后优先归入兼容的 `planning` 版本；没有满足产品、仓库、分支和策略条件的版本才新建版本 |
| 交付终点 | `ready_for_release`；不创建、启动或推进部署 |
| 允许写入的代码范围 | 策略编辑页、协同 DAG、协同客户端类型、对应前端测试、帮助文档和真实截图 |
| 既有未提交改动 | 演示开始前必须记录基线；任何非允许路径进入暂存区均停止并交由真人决定 |
| Git | 只能写入当前研发分支；受保护分支和未声明路径禁止写入 |

## 目标行为与验收标准

### 策略页面

1. 研发执行策略编辑器可维护非空、去重的仓库可信域和工具可信域列表。
2. 保存后这两个列表进入 `experience_reuse_config`，通过策略版本乐观锁形成新的不可变快照。
3. 页面不展示 Runner Token、Git 凭据或其他秘密；可信域仅是受控标识。

### 协同工作项

1. 客户端读取工作项的 `input_contract`，当存在 `approved_role_experience_context` 时显示引用数量和每条经验的标识、版本及证据引用数量。
2. 没有经验时不渲染空的经验区域，原有 DAG 状态、依赖、风险和优先级展示不回退。
3. 页面只展示引用，不将经验视作策略或允许其改变冻结的岗位、预算、质量门禁或交付终点。

### 测试与帮助

1. 前端测试覆盖策略 payload、保存后可信域保留、工作项有/无经验两种渲染状态。
2. 帮助中心说明可信域、独立审核、批准经验的受控复用和 `ready_for_release` 不部署边界，并使用真实本地截图。
3. 提交前通过相关 Vitest、TypeScript 类型检查、帮助资产校验和真实登录页面检查。

## 演示角色与策略

| 岗位 | 执行主体 | 职责 |
| --- | --- | --- |
| 产品经理 | AI 数字员工 | 需求评估、验收条件和工作项建议 |
| 开发 | 本地 Codex AI 员工 | 在允许路径内实现、运行局部验证、提供证据 |
| 测试 | AI 数字员工 | 编写/更新测试、执行 Vitest 和类型检查、提交测试证据 |
| 文档 | AI 数字员工 | 更新帮助说明、生成真实截图并形成知识沉淀材料 |
| 项目/研发负责人 | 真人账号 | 处理需要人工确认的范围、代码审核或 Git 门禁 |
| 知识与经验审核人 | 与证据生产者隔离的真人账号 | 批准/拒绝项目知识候选和岗位经验候选 |

策略冻结前的准备条件：

- 本地 Codex Runner 状态健康、具备声明的工作区和工具可信域；如果 Runner 心跳、能力或 Git 受权检查失败，演示停在预检，不伪造执行结果。
- API 与执行 Worker 均设置 `RD_ROLE_EXPERIENCE_ENABLED=true` 后重启，使第一轮岗位反馈可以生成经验候选。
- 策略的交付终点为 `ready_for_release`，启用远程提交证据和独立代码审核，不启用部署。

## 两轮真实协同流程

```text
第一轮：功能迭代
需求 -> 评估 -> 组版 -> 冻结策略/范围 -> DAG
     -> AI 开发 + AI 测试 + AI 文档 -> 真人审核 -> 远程提交
     -> ready_for_release -> 项目知识候选 + 岗位经验候选 -> 独立批准

第二轮：复用验证
小型兼容需求 -> 新的范围/策略快照 -> DAG 规划
             -> 注入已批准经验引用 -> 页面显示引用及证据
```

### 第一轮：真实功能交付

1. 产品负责人在需求管理中创建并提交本需求，包含上述验收标准与受限代码范围。
2. 系统执行岗位评估；缺少信息或高风险结论进入人工决策，只有接受后才能组版。
3. 系统优先选择 AI Brain 的兼容规划中版本；如没有可用版本，才创建新的规划版本。
4. 从版本总览启动研发协同。平台冻结需求修订、仓库/分支、策略快照、岗位席位和验收标准，并生成 DAG。
5. 开发、测试和文档工作项按依赖派发。所有代码、测试和文档产出必须带 attempt、执行和质量证据。
6. 真人审核人确认代码审查与远程提交门禁。确认后只推送声明文件的独立提交到 `origin/codex/rd-collaboration-v2`。
7. 测试、代码审查、远程 commit 和交付证据齐备后，版本进入 `ready_for_release`；协同不会进入部署域。
8. 完成产物生成项目知识库沉淀候选，知识审核人批准后形成可检索知识文档及审计事件。
9. 每次执行、审核、返工和门禁生成不可变岗位反馈；开启 P1 后生成 `pending` 岗位经验候选。与任一来源生产者隔离的审核人批准候选。

### 第二轮：受控复用验证

1. 在第一轮已提交的新策略页面中填写与 AI Brain 仓库、执行器匹配的可信域，并保存为新策略版本。
2. 创建一个同产品、同策略 Schema、同可信域且风险和岗位兼容的小型验证需求。
3. 评估和组版后启动新的协同运行。规划服务仅在可信域、范围、最小置信度、最大条数、Token 上限、时效和策略兼容性同时满足时，把批准经验附加到工作项输入契约。
4. 在 DAG 页面查看经验引用和证据；同时通过运行/工作项详情及审计记录确认该经验没有修改策略、权限、预算或交付终点。

## 人工决策、失败与恢复

| 情况 | 平台行为 | 演示处理 |
| --- | --- | --- |
| Runner、模型或 Git 预检失败 | 不启动或暂停受影响分支 | 修复配置后从原阶段继续，不伪造成功 |
| 超出允许路径 | 阻止暂存/提交 | 真人决定扩大范围或取消该项 |
| 质量门禁失败 | 工作项返工或等待人工决策 | 保留原 attempt、证据和失败原因后再领取 |
| 远程提交/代码审核门禁 | 等待独立真人确认 | 确认后继续，拒绝则返工 |
| 知识或经验审核 | `pending` 保持不可自动复用 | 仅独立审核人批准后可进入第二轮上下文 |
| 策略/范围变化 | 新快照或范围变更流程 | 不原地重写活动运行；必要时以新运行代次继续 |

## 演示完成证据

演示完成必须同时取得以下证据：

1. 已接受需求、评估结论、目标版本、冻结 scope/策略快照和工作项 DAG。
2. 受限路径内的真实代码差异、Vitest 与类型检查结果、真人审核记录。
3. 指向 `codex/rd-collaboration-v2` 的远程 commit SHA 与可信 Git 对账证据。
4. 版本 `ready_for_release` 结论，且不存在部署任务或部署执行记录。
5. 已批准的项目知识沉淀记录、生成文档/索引状态及审计事件。
6. 已批准的岗位经验记录、来源反馈和隔离审核记录。
7. 第二轮工作项中可见的经验引用与证据，及其受控匹配条件。
