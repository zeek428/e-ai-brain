# 运维部署独立菜单设计

## 背景

运维部署已经具备部署单创建、启动、成功、失败、回滚和取消能力，但当前入口、列表和操作均混在“运营治理 / 日志监控”页面。部署属于交付状态推进和高风险操作，日志监控则用于工程指标、构建发布和线上日志观测，两者的页面职责和权限边界不同。

## 目标

- 新增“运营治理 / 运维部署”菜单，路径为 `/governance/deployments`。
- 运维部署页面只展示部署单和部署操作，日志监控不再展示部署单或“发起部署”。
- 菜单访问要求 `deployment.read`；页面操作分别要求 `deployment.create`、`deployment.execute`、`deployment.cancel`。
- 部署数据继续复用现有 PostgreSQL 事实表、部署接口和服务端分页 read model，不新增重复业务状态。
- 版本驾驶舱、需求全链路和帮助中心中的部署入口跳转到新页面；Jenkins 发布记录仍跳转日志监控。

## 方案比较

### 方案 A：为同一个日志监控页面增加第二个菜单别名

改动最少，但两个菜单展示相同内容，部署和日志职责仍然混杂，权限按钮也难以清晰收口，不采用。

### 方案 B：复制现有日志监控页面形成部署页面

可以快速获得独立页面，但会复制部署表单、状态动作和数据加载逻辑，后续修改容易出现行为不一致，不采用。

### 方案 C：拆出独立部署页面并复用现有接口和 read model

运维部署页面只持有部署相关状态和表单；日志监控保留三类观测数据。部署列表继续调用 `/api/devops/operational-metrics?category=运维部署` 获得服务端分页数据，写操作继续调用 `/api/devops/deployments` 系列接口。该方案边界清楚且不复制后端状态，采用此方案。

## 页面与权限

菜单资源：

- 编码：`governance.deployments`
- 名称：`运维部署`
- 路径：`/governance/deployments`
- 父菜单：`governance`
- 菜单权限：`deployment.read`

默认角色授权遵循已有权限定义：管理员、产品负责人、研发负责人、测试负责人、测试人员、发布负责人拥有部署读取权限，因此获得该菜单；只有拥有 `deployment.create` 的用户显示“发起部署”，只有拥有 `deployment.execute` 的用户显示启动/完成/失败/回滚，只有拥有 `deployment.cancel` 的用户显示取消。

## 数据流

1. 页面以固定分类“运维部署”查询运营统一 read model，服务端完成产品 scope、筛选、排序和分页。
2. 创建部署单时继续读取当前用户可见的产品、版本和测试完成/待发布需求，并调用现有创建接口。
3. 启动、完成、失败、回滚和取消继续调用现有动作接口；动作完成后刷新部署列表。
4. 日志监控查询增加排除分类参数，确保分页总数和列表内容均不包含部署单。
5. `version_id` 或 `deployment_id` 深链作为页面初始关键字，便于从版本驾驶舱和全链路定位部署记录。

## 错误与边界

- 后端继续作为权限和产品 scope 的最终校验层；前端按钮隐藏只用于减少无权限误操作。
- 列表加载失败沿用管理列表错误提示和 `trace_id` 展示。
- 无部署记录时展示真实空列表，不构造占位数据。
- 现有 `/governance/devops` 路由保持不变，避免破坏日志监控书签；部署深链更新到新路由。

## 验收

- 管理员和发布负责人可在“运营治理”下看到“运维部署”。
- 仅具备 `deployment.read` 的测试人员可查看部署单，但看不到高风险操作按钮。
- 日志监控不展示部署分类和发起部署按钮。
- 运维部署页面可创建部署单并按权限执行状态动作。
- 版本驾驶舱和需求全链路中的部署链接指向 `/governance/deployments`。
- 后端 RBAC、前端页面、类型检查、帮助中心检查和真实浏览器流程通过。
