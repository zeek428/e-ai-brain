# AI 自动开发流程使用指南

本指南说明如何利用本文档模板体系，实现 AI 辅助的自动化开发流程。

## AI Brain 项目约定

本指南中的 `{feature}` 流程是通用模板说明。当前 AI Brain 项目后续迭代以项目级文档为维护源：产品范围更新 [01-prd/enterprise-ai-brain/prd.md](../01-prd/enterprise-ai-brain/prd.md)，技术设计更新 [02-specs/enterprise-ai-brain/spec.md](../02-specs/enterprise-ai-brain/spec.md)，接口更新 [02-specs/enterprise-ai-brain/api.md](../02-specs/enterprise-ai-brain/api.md)，验收测试更新 [02-specs/enterprise-ai-brain/test-case.md](../02-specs/enterprise-ai-brain/test-case.md)。只有明确需要独立功能包时，才复制 `_template/` 新建 feature 级文档。

## 核心理念

**文档即需求，AI 即执行者**

这套文档体系专为 AI 原生开发设计：
- 结构化文档让 AI 快速理解项目上下文
- 标准化模板减少沟通成本
- 完整的信息链支持自动化开发

## AI 阅读顺序

```
1. CLAUDE.md          → 理解项目整体上下文
2. glossary.md        → 理解专业术语
3. 01-prd/{feature}/prd.md   → 理解需求
4. 02-specs/{feature}/spec.md → 理解技术方案
5. 02-specs/{feature}/api.md  → 理解接口定义
6. 02-specs/{feature}/test-case.md → 理解验收标准
```

## 文档作用详解

### CLAUDE.md - AI 入口点

**作用**: 让 AI 在 30 秒内理解项目

**必填内容**:
- 项目概述（一句话说明项目是什么）
- 技术栈（语言、框架、数据库）
- 常用命令（构建、测试、部署）
- 目录结构说明
- 禁止事项

**AI 能做什么**:
- 快速定位代码位置
- 理解架构决策
- 遵循项目规范

### PRD 文档 - 需求定义

**作用**: 告诉 AI "做什么"

**编写方式**: 人工提出需求想法 → AI 生成 PRD → 人工确认验收 → AI 生成高保原型

**核心要素**:
| 要素 | 作用 | AI 理解方式 |
|------|------|-------------|
| 用户故事 | 定义功能目标 | 提取实现目标 |
| 验收标准 | 定义完成定义 | 生成测试用例 |
| In/Out Scope | 定义边界 | 避免过度实现 |
| 高保原型 | 可视化需求 | 理解 UI 交互 |

**示例对话**:
```
用户: 我需要一个用户注册功能，支持手机号和邮箱注册
AI: 我已根据您的需求生成 PRD，包含：
    1. 手机号注册流程（验证码发送、校验）
    2. 邮箱注册流程（邮箱验证）
    3. 密码加密存储
    请确认验收标准是否符合预期
用户: 验收标准没问题，生成高保原型
AI: 已生成 01-prd/user-registration/prototype/index.html，包含注册页面 UI 布局和交互流程
```

### Spec 文档 - 技术设计

**作用**: 告诉 AI "怎么做"

**核心要素**:
| 要素 | 作用 | AI 理解方式 |
|------|------|-------------|
| 设计决策 | 技术选型理由 | 遵循既定方案 |
| 架构图 | 系统结构 | 理解模块关系 |
| 数据库设计 | 数据结构 | 生成迁移文件 |
| API 设计 | 接口规范 | 生成接口代码 |
| 测试策略 | 质量保障 | 生成测试用例 |

**示例对话**:
```
用户: 根据 02-specs/user-management/spec.md 实现数据层
AI: 根据技术规格，我需要：
    1. 创建 users 表（遵循 spec 中的表结构设计）
    2. 实现 UserRepository 接口
    3. 使用 bcrypt 进行密码哈希（符合设计决策）
```

### API 文档 - 接口契约

**作用**: 定义前后端交互规范

**核心要素**:
| 要素 | 作用 |
|------|------|
| 请求/响应格式 | 数据结构定义 |
| 错误码 | 异常处理逻辑 |
| 认证方式 | 安全实现 |
| 示例 | 快速理解 |

**AI 使用场景**:
- 生成 Controller 代码
- 生成前端 API 调用
- 生成 Mock 数据
- 生成 API 测试

### Test Case 文档 - 验收标准

**作用**: 定义 "完成" 的标准

**核心要素**:
| 要素 | 作用 |
|------|------|
| 前置条件 | 测试环境准备 |
| 测试步骤 | 执行路径 |
| 预期结果 | 断言依据 |
| 边界测试 | 边界条件覆盖 |

**AI 使用场景**:
- 生成单元测试
- 生成集成测试
- 生成 E2E 测试
- 验证实现完整性

## AI 开发工作流

### 阶段一：需求分析

```bash
# 1. AI 阅读项目上下文
用户: 请阅读 CLAUDE.md，告诉我这个项目的技术栈

# 2. 人工提出需求想法
用户: 我需要一个用户登录功能，支持手机号+验证码登录

# 3. AI 生成 PRD 文档
AI: 我已根据您的需求生成 01-prd/user-login/prd.md，包含：
    - 用户故事：作为用户，我希望通过手机验证码登录...
    - 验收标准：验证码有效期5分钟，每日发送上限5次...
    请确认是否符合预期

# 4. 人工确认验收
用户: 验收标准通过

# 5. AI 生成高保原型
用户: 根据 PRD 生成高保原型
AI: 已生成 01-prd/user-login/prototype/index.html，包含：
    - 登录页面 UI 布局
    - 验证码输入交互
    请确认原型效果
```

### 阶段二：技术设计

```bash
# 1. AI 基于需求生成技术方案
用户: 根据 PRD 生成 02-specs/user-login/spec.md

# 2. 人工 Review 并修改
用户: 数据库设计需要增加 deleted_at 字段

# 3. AI 更新项目级技术规格
用户: 更新 spec.md，所有表增加软删除字段
```

### 阶段三：代码实现

```bash
# 1. AI 按照项目级文档实现
用户: 根据 02-specs/user-login/ 实现 Controller 层

# 2. AI 自行检查是否符合设计
AI: 已实现 UserController，包含：
    - POST /api/login（符合 API 文档定义）
    - 错误处理（符合错误码规范）

# 3. AI 生成测试
用户: 根据 test-case.md 生成单元测试
```

### 阶段四：验证与交付

```bash
# 1. AI 运行测试
用户: 运行所有测试
AI: 正在执行测试...
    - 单元测试: 45/45 通过
    - 集成测试: 12/12 通过
    - E2E 测试: 8/8 通过
    覆盖率: 85%

# 2. AI 生成测试报告
用户: 生成测试报告
AI: 已生成测试报告，包含：
    - 测试执行汇总（通过/失败/跳过）
    - 覆盖率分析
    - 性能指标
    - 风险项提示
    报告位置: 02-specs/user-login/test-report.md

# 3. 人工复核测试报告
用户: 复核测试报告，覆盖率达标，可以验收

# 4. 人工测试验收
用户: 我来执行手工验收测试
    - 验证核心业务流程
    - 验证边界条件
    - 验证 UI 交互
    验收通过

# 5. AI 检查代码规范
用户: 检查代码是否符合 06-standards/coding-standards.md

# 6. AI 更新文档
用户: 更新 changelog.md，记录本次变更
```

## 文档编写最佳实践

### 对 AI 友好的文档特征

| 特征 | 说明 | 示例 |
|------|------|------|
| 结构化 | 使用固定章节 | PRD 必含用户故事、验收标准 |
| 具体化 | 避免模糊表述 | ✅ "密码长度 8-20 位" ❌ "密码要安全" |
| 可执行 | 提供命令示例 | `npm run test:unit` |
| 有边界 | 明确 In/Out Scope | 防止 AI 过度实现 |
| 有示例 | 提供参考代码 | API 文档含 curl 示例 |

### 避免的文档问题

| 问题 | 后果 | 改进 |
|------|------|------|
| 信息分散 | AI 需多次追问 | 集中在对应文档 |
| 缺少上下文 | AI 理解偏差 | CLAUDE.md 补充背景 |
| 模糊表述 | AI 实现不准确 | 使用量化指标 |
| 过时未更新 | 代码与文档不一致 | PR 流程包含文档检查 |

## 常见场景示例

### 场景 1：新功能开发

```
1. 复制 01-prd/_template/prd.md → prd/new-feature/prd.md
2. 填写需求文档
3. 让 AI 生成 02-specs/new-feature/spec.md
4. Review 并完善技术设计
5. AI 按设计实现代码
6. AI 生成测试并验证
```

### 场景 2：Bug 修复

```
1. 在 02-specs/{feature}/test-case.md 添加失败测试用例
2. 让 AI 分析原因
3. AI 定位问题代码
4. AI 修复并验证测试通过
5. 更新 changelog.md
```

### 场景 3：代码重构

```
1. 在 04-decisions/ 创建 ADR 文档说明重构原因
2. 更新 02-specs/{feature}/spec.md 中的设计
3. AI 根据新设计重构代码
4. AI 确保测试全部通过
```

## AI 能力边界

**AI 擅长**:
- 理解结构化文档
- 按照规范生成代码
- 执行明确的指令
- 生成测试用例

**需要人工**:
- 业务决策
- 架构权衡
- 需求澄清
- 最终验收
