# 需求交付

## 需求管理

需求管理用于登记、审批、关闭需求，并从需求生成 AI 研发任务。

### 核心流程

1. 创建需求并选择产品、版本、优先级和负责人。
2. 补充需求背景、业务目标、验收标准和关联材料。
3. 提交审批，审批通过后可生成 AI 研发任务。
4. 通过“全链路”查看需求、任务、评审、Bug、知识和审计之间的关系。

viewer 只能查看授权产品范围内的需求，不能创建、审批、关闭或生成任务。

## 研发任务

研发任务用于跟踪 AI Task 的执行状态、人工确认点和写回结果。

常见状态：

- `running`：AI 正在执行。
- `waiting_review`：等待人工确认。
- `waiting_more_info`：需要补充上下文。
- `completed`：任务完成。
- `failed`：执行失败，需要查看执行诊断。

任务详情中优先查看输出摘要、确认点、执行诊断、关联知识引用和审计记录。确认通过后才会进入写回或合入动作。

## 研发执行器策略

研发执行器策略决定不同任务类型如何选择 AI 执行器、知识上下文和代码提交方式。

配置建议：

- 按任务类型、产品、优先级和风险等级匹配策略。
- 默认使用人工确认的代码提交方式。
- 自动提交只适合边界清晰、测试充分、回滚成本可控的任务。

## 迭代版本

迭代版本用于把需求、任务、Bug 和发布记录归属到同一版本。版本级全链路可以查看该版本下的交付范围和风险。暂无需求的版本会展示版本级空状态，而不是直接报错。

需求开发和测试完成后，需求会进入“待发布”。版本推进到已发布前，需要在“运营治理 / 运维部署”中发起部署单并登记部署结果；部署成功后，关联需求进入“已发布”。部署失败或回滚后，需求会回到“待发布”，并生成“部署失败”来源的 Bug 供后续修复。

## Bug 管理

Bug 管理用于登记、分诊、推进和追踪缺陷生命周期。

常用操作：

1. 按产品、版本、状态、严重级别和来源筛选 Bug。
2. 查看 Bug 详情、关联任务和全链路。
3. 具备写权限的用户可以登记、编辑、批量处理或推进 AI 任务。

viewer 只展示 Bug 列表与全链路入口，不展示登记、编辑、删除、批量处理和行选择。

## 全链路

全链路用于追踪需求、版本、任务、评审、Bug、知识、代码巡检和审计之间的上下游关系。独立代码巡检报告或缺少需求上下文的主体可能不会展示需求全链路入口。
